# Инструкция для менеджеров
## Логика взаимодействия с Bitrix24

---

## Общее понимание системы

Middleware сервис - это автоматический мост между Shopify (ваш интернет-магазин) и Bitrix24 (система управления продажами). Он работает в фоновом режиме и синхронизирует данные между двумя системами.

---

## Как работает синхронизация заказов

### 📥 Shopify → Bitrix24 (Автоматически)

**Что происходит:**
1. Клиент оформляет заказ на сайте или в магазине
2. Shopify отправляет уведомление в Middleware
3. Middleware автоматически создаёт сделку в Bitrix24

**Какие данные переносятся:**
- ✅ Номер заказа и дата
- ✅ Все товары из заказа (название, количество, цена)
- ✅ Данные покупателя (имя, телефон, email)
- ✅ Адрес доставки
- ✅ Сумма заказа
- ✅ Статус оплаты
- ✅ Способ доставки

**Воронки продаж (куда попадает заказ):**
- **"Stock (in the shop)"** - заказы из физического магазина (POS)
- **"Stock (site)"** - заказы с сайта
- **"Pre-order (in the shop)"** - предзаказы из магазина
- **"Pre-order (site)"** - предзаказы с сайта

**Статусы сделок:**
- **NEW** - заказ создан, ожидает оплаты
- **WON** - заказ оплачен
- **C2:PREPARATION** - заказ частично возвращён, но ещё активен
- **LOSE** - заказ отменён или полностью возвращён

### 📤 Bitrix24 → Shopify (Автоматически)

**Что происходит:**
1. Менеджер создаёт сделку в Bitrix24 с товарами
2. Если у сделки нет связанного заказа в Shopify, Middleware автоматически создаёт технический заказ
3. Заказ в Shopify помечается меткой: **BITRIX:{ID сделки}**

**Зачем это нужно:**
- Для правильного учёта товаров на складе в Shopify
- Чтобы остатки товаров были актуальными в обеих системах
- Резервирует товары при создании сделки в Bitrix24

**Автоматическая отмена технического ордера:**
- Если сделка переводится в стадию **LOSE** (отменена/проиграна), технический ордер автоматически отменяется в Shopify
- Товары возвращаются в инвентарь (restock)
- Это позволяет корректно обрабатывать отменённые заказы

**Защита от циклов (loop guard):**
- Если заказ в Shopify был изменён из Bitrix (например адрес/доставка/отмена), Middleware добавляет тег **`BitrixUpdated`**
- Shopify webhooks с тегом `BitrixUpdated` (и/или с маркером `middleware.last_write=bitrix`) **пропускаются**, чтобы избежать бесконечной синхронизации

## Работа с товарами в Bitrix24

### Как товары связываются между системами?

**Основной способ связи:**
- Каждый товар в Bitrix24 имеет поле **CODE** (это SKU из Shopify)
- Каждый товар имеет поле **XML_ID** (это уникальный ID варианта товара из Shopify - variant_id)

**Важно понимать:**
- Если товар не найден по SKU или variant_id, он не будет добавлен в сделку
- Товары должны быть правильно настроены в обеих системах

### Что делать, если товар не добавился в сделку?

1. **Проверьте SKU товара** в Bitrix24 (поле CODE)
2. **Проверьте XML_ID** товара (должен совпадать с variant_id из Shopify)
3. **Проверьте логи** в веб-интерфейсе Middleware
4. Если товар отсутствует в Bitrix24 - создайте его через UI Middleware

## Статусы заказов и их значение

### Статусы оплаты (Financial Status)

| Статус в Shopify | Статус в Bitrix24 | Описание |
|-----------------|-------------------|----------|
| **paid** | WON | Заказ полностью оплачен |
| **pending** | NEW | Заказ ожидает оплаты |
| **partially_paid** | NEW | Заказ частично оплачен, ожидает доплаты |
| **refunded** | LOSE | Заказ полностью возвращён |
| **partially_refunded** | C2:PREPARATION | Заказ частично возвращён, но ещё активен |
| **cancelled** | LOSE | Заказ отменён |
| **voided** | LOSE | Заказ аннулирован |

### Статусы выполнения (Fulfillment Status)

| Статус в Shopify | Что происходит в Bitrix24 |
|-----------------|---------------------------|
| **fulfilled** | Статус выполнения обновляется |
| **partial** | Частично выполнен, статус обновляется |
| **unfulfilled** | Ожидает выполнения |

## Специальные случаи

### 1. Технические заказы

**Что это:**
- Заказы, созданные автоматически из Bitrix24 в Shopify
- Помечены меткой: **BITRIX:{ID сделки}**

**Когда создаются:**
- Когда менеджер создаёт сделку в Bitrix24 с товарами
- Но у сделки ещё нет связанного заказа в Shopify
- Система автоматически резервирует товары в Shopify

**Когда отменяются:**
- Автоматически отменяются, когда сделка переводится в стадию **LOSE** (отменена/проиграна)
- Товары автоматически возвращаются в инвентарь

**Что делать:**
- Ничего! Это нормальная работа системы
- Такие заказы нужны для учёта товаров и резервирования остатков
- Если сделка отменена - технический ордер отменится автоматически

### 2. Предзаказы (Pre-order)

**Как определить:**
- Заказы с тегом **"preorder"** в Shopify
- Попадают в воронку "Pre-order"

**Особенности:**
- Обрабатываются так же, как обычные заказы
- Но попадают в отдельную воронку для отчётности

### 3. Заказы из магазина (POS)

**Как определить:**
- Заказы с `source_name = "pos"` в Shopify
- Попадают в воронку "Stock (in the shop)"

**Особенности:**
- Отличаются от заказов с сайта
- Нужны для правильной отчётности по каналам продаж

### 4. Сертификаты

**Особенности:**
- Подарочные сертификаты обрабатываются специальным образом
- Имеют свою логику связывания (по variant_id, а не по SKU)
- Автоматически создаются и обновляются в Bitrix24

## Что делать при проблемах?

### Проблема: Заказ не появился в Bitrix24

**Проверьте:**
1. Откройте веб-интерфейс Middleware
2. Посмотрите список событий из Shopify
3. Найдите ваш заказ
4. Проверьте, есть ли ошибки

**Решение:**
- Если заказ есть в списке, но не отправлен - выберите его и нажмите "Отправить в Bitrix"
- Если есть ошибки - проверьте логи

### Проблема: Товары не добавились в сделку

**Проверьте:**
1. Есть ли товар в каталоге Bitrix24?
2. Правильно ли указан SKU (CODE) у товара?
3. Правильно ли указан XML_ID (variant_id)?

**Решение:**
- Если товара нет - создайте его через UI Middleware
- Если товар есть, но не связывается - проверьте SKU и XML_ID

### Проблема: Статус не обновился

**Проверьте:**
1. Откройте веб-интерфейс Middleware
2. Посмотрите события из Bitrix24
3. Проверьте, отправлено ли событие в Shopify

**Решение:**
- Если событие есть, но не отправлено - выберите его и нажмите "Отправить в Shopify"
- Если событие отсутствует - возможно, webhook из Bitrix24 не настроен

## Работа с веб-интерфейсом

### Основные разделы:

1. **Shopify → Bitrix Events**
   - Все заказы, которые пришли из Shopify
   - Можно просмотреть детали каждого заказа
   - Можно вручную отправить заказ в Bitrix24

2. **Bitrix → Shopify Events**
   - Все события, которые пришли из Bitrix24
   - Можно просмотреть детали каждого события
   - Можно вручную отправить событие в Shopify

3. **Success Operations**
   - Успешно выполненные операции
   - История синхронизаций

4. **Управление товарами**
   - Создание товаров по категориям
   - Синхронизация остатков
   - Обновление информации о товарах

### Как использовать:

1. **Просмотр заказов:**
   - Откройте раздел "Shopify → Bitrix Events"
   - Вы увидите список всех заказов
   - Нажмите на заказ, чтобы увидеть детали

2. **Ручная отправка:**
   - Выберите заказы чекбоксами
   - Нажмите кнопку "Отправить в Bitrix"
   - Дождитесь результата

3. **Просмотр логов:**
   - Нажмите кнопку "Скачать логи"
   - Откроется файл со всеми операциями
   - Можно найти любую операцию по времени или ID

## Важные моменты для менеджеров

### ✅ Что можно делать:

- Создавать сделки в Bitrix24 - они автоматически синхронизируются
- Менять статусы сделок - они обновляются в Shopify
- Добавлять товары в сделки - они учитываются в Shopify
- Просматривать логи для проверки операций

### ❌ Чего не нужно делать:

- Не нужно вручную создавать/дублировать заказы в Shopify, если сделка уже есть в Bitrix24
- Не нужно дублировать данные - система сделает это автоматически
- Не нужно менять технические поля (SKU, XML_ID) без необходимости

## Тест‑кейсы (как НУЖНО / как НЕ НУЖНО)

### ✅ Как НУЖНО (основные проверки)

1. **POS продажа → Bitrix**
   - Создайте заказ в Shopify POS (у заказа `source_name="pos"`, без preorder‑тегов)
   - Ожидаемо: сделка создаётся в Bitrix в воронке **Stock (in the shop)** (category_id=0)

2. **Сайт заказ → Bitrix**
   - Создайте заказ на сайте
   - Ожидаемо: сделка создаётся в воронке **Stock (site)** (category_id=2)

3. **Pre‑order (site/in the shop)**
   - Создайте заказ с preorder‑тегом в Shopify
   - Ожидаемо: POS pre‑order → **Pre-order (in the shop)** (category_id=4), сайт pre‑order → **Pre-order (site)** (category_id=8)

4. **Адрес доставки из Bitrix → Shopify**
   - В сделке Bitrix измените поле адреса доставки `UF_CRM_1742037435676` (и при необходимости стоимость доставки `UF_CRM_67BEF8B2AA721`)
   - Ожидаемо: в Shopify обновится `shipping_address` (и `shipping_lines` при изменении цены доставки)

5. **Отмена сделки LOSE → отмена заказа в Shopify**
   - Переведите сделку в стадию **LOSE**
   - Ожидаемо: заказ в Shopify отменяется (restock), и Middleware ставит тег **BitrixUpdated** для защиты от loop

6. **Стадия Delivery → обновление доставки в Shopify**
   - Для воронки **Stock (site)** (category_id=2) переведите сделку в стадию **C2:EXECUTING** (Delivery)
   - Ожидаемо: Middleware создаст/обновит fulfillment в Shopify (чтобы отражалось “в доставке”)

### ❌ Как НЕ НУЖНО (чтобы не создавать путаницу)

- **Не трогайте вручную теги заказа в Shopify**: `BITRIX:{dealId}` и `BitrixUpdated` (они нужны для связки и loop guard)
- **Не меняйте вручную поле связи `UF_CRM_1742556489` в Bitrix** (это Shopify Order ID)
- **Не делайте “двойные” изменения одновременно** (например, адрес в Shopify и адрес в Bitrix в одно и то же время) — выбирайте одну систему как источник правки

### ⚠️ На что обратить внимание:

- Если товар не добавился в сделку - проверьте, есть ли он в каталоге Bitrix24
- Если заказ не синхронизировался - проверьте логи в веб-интерфейсе
- Если статус не обновился - возможно, нужно вручную отправить событие

## Важная информация об ограничениях

### ⚠️ Инвентаризация товаров

**Текущее ограничение:**
- Система обрабатывает только товары с количеством на складе **больше нуля** (`qty > 0`)
- Товары с нулевым остатком **не синхронизируются** автоматически

**Что это значит для вас:**
- Если товар закончился на складе (остаток = 0), он не будет автоматически обновляться в Bitrix24
- При массовом создании товаров через UI Middleware обрабатываются только товары с остатком > 0
- Это сделано для оптимизации и избежания обработки неактуальных товаров

**Что делать:**
- Если нужно обновить товары с нулевым остатком, это нужно делать вручную через Bitrix24
- Или дождаться, когда товар появится на складе (qty > 0), тогда он автоматически синхронизируется

### ⚠️ Обновление полей товаров (Size и другие)

**Известная проблема:**
- При автоматическом обновлении товаров в Bitrix24 могут возникать проблемы с обновлением некоторых полей:
  - **Size (Размер)** - поле PROPERTY_98 не всегда обновляется корректно
  - **Color (Цвет)** - поле PROPERTY_106 может требовать ручного обновления
  - Другие пользовательские поля могут не синхронизироваться автоматически

**Почему это происходит:**
- Некоторые поля в Bitrix24 требуют специальной обработки (например, Size требует поиска ID значения, а не текста)
- При массовом обновлении некоторые поля могут пропускаться из-за ограничений API

**Что делать:**
- **Для критически важных товаров:** обновляйте поля вручную через интерфейс Bitrix24
- **При массовом обновлении:** проверяйте, что все поля обновились корректно
- **Если поле не обновилось:** используйте ручное обновление через Bitrix24

**Временное решение:**
- Для товаров, где важно иметь актуальные Size и другие поля, рекомендуется:
  1. Сначала создать товар через Middleware
  2. Затем вручную проверить и обновить поля Size, Color и другие в Bitrix24
  3. После этого товар будет корректно связываться с заказами

**Планируется исправить:**
- Улучшение логики обновления полей Size и других пользовательских свойств
- Более надёжная синхронизация всех полей товаров
- Автоматическая проверка и повторная попытка обновления при ошибках
